Enhancement Added: List Alphabet

External behavior

Start of Extract [from: User Guide]

Searching all persons whose name starts with a certain alphabet: listalp

Finds persons whose names start with the given alphabet.
Format: listalp ALPHABET

  • The search is case sensitive.

End of Extract

Justification

For a forgetful user, when number of contacts stored in Contact+ Pro is very large, it would be extremely difficult to find a certain contact if he could not recall the full name of the contact. The list alphabet command aims to alleviate the difficulty: instead of searching by the full name or other attributes which are hard to remember, the user can search by the first alphabet of the name of the contact, and thus search for the desired contact more easily within a much narrower range.

Implementation

Start of Extract [from: Developer Guide]

List Persons whose names start with certain alphabet

List alphabet function is designated for frequent users who have a large number of contacts saved in the addressbook and could not recall the full name of the person he wants to find.

List alphabet function is implemented in ListAlphabetCommand under logic component. When parsing the ListAlphaCommand in ListAlphaCommandParser, instead of passing the input argument to the ListAlphabetCommand, the parser invokes a method called NameStartsWithAlphabetPredicate.

NameStartsWithAlphabetPredicate is a class created under model component, which checks for any persons whose name starts with the character same as the input argument, and returns a predicate to be passed into ListAlphabetCommand.

ListAlphabetCommand reads the predicate and update filtered person list according to the predicate. After updating person list, Command result is returned with size of the updated filtered person list.

Design Consideration

Aspect: Way of comparing input argument and name
Alternative 1 (current choice): Comparing the first character of input argument and first character of a person’s full name.
Pros: Easy to implement.
Cons: Only the first input alphabet is effective, and unable to search for multiple alphabets.
Alternative 2: Use regex to restrict input argument format, compare each single input character to get a full list of persons.
Pros: Allows user to search for multiple alphabets at once.
Cons: User might need to repeatedly type the same command if the input format is different from preset regex.

End of Extract

Enhancement Added: Locate

External behavior

Start of Extract [from: User Guide]

Finding the route from the specified address to a person in the addressbook: locate

Finds the route from the given address to the address of the specified contact.
Format: locate INDEX a/ADDRESS

Example:

  • locate 1 a/clementi
    Displays the route from clementi to the address of the first person in the current list.

End of Extract

Justification

For a user who likes to visit his friends, navigation function is a crucial part of the address book his is using. With the locate command, the user can easily find out the route from any location to the location of any of his friends or relatives.

Implementation

Start of Extract [from: Developer Guide]

Locate: Navigate to a person’s address

Locate function allows the user to find the route from the address entered to a person’s address, and the route displayed is the recommended by the Google Maps. The parameters to be entered by the user are the index of the person to be located, and the starting address with the address prefix in the front.

During the implementation, the LocateCommandParser parses the input argument and extracts the index of the person and the starting address, and passes these values to a new LocateCommand object. The LocateCommand object obtains the address based on the index, and raises an event called BrowserPanelLocateEvent with the 2 addresses.

Below shows the implementation of BrowserPanelLocateEvent:

public class BrowserPanelLocateEvent extends BaseEvent {

    private final String start;
    private final String end;

    public BrowserPanelLocateEvent(String start, String end) {
        this.start = start;
        this.end = end;
    }

    @Override
    public String toString() {
        return this.getClass().getSimpleName();
    }

    public String getEndAddress() {
        return end;
    }

    public String getStartAddress() {
        return start;
    }
}

The sequence diagram shown below shows how locate command works with input locate 1 a/clementi:

Only one index and one address with address prefix are accepted. If input format is different, an invalid input message will be prompted. If the index given is wrong, an invalid person displayed index message will be prompted.

Design Considerations

Aspect: what the start and end addresses should be
Alternative 1 (current choice): The address input by the user is regarded as starting address by default, the address of the person in Contact+ Pro is set to end address.
Pros: Easier implementation
Command usage is more straight forward for the user.
Cons: User is not able to navigate from a person’s address to a new location.
Alternative 2: The user decides which address is the start address and which is the end address. Pros: User has more freedom when using this locate command.
Cons: More inputs are expected from the user, causing the command to be more complicated.

End of Extract
